> 대화형 아이디어 구체화 → 이슈 생성 | context_tier: 200k

사용자와 대화하며 아이디어를 구체화하고, 실행 가능한 GitHub 이슈로 변환합니다.

인자: `$ARGUMENTS` (선택, 있으면 해당 주제로 시작. 없으면 질문부터 시작)

## Step 1: 아이디어 탐색 — 질문 공세

### 인자가 없는 경우

코드베이스를 빠르게 파악한 뒤 (package.json, 디렉토리 구조, README), 구체적인 질문으로 시작한다:

```
프로젝트를 살펴봤습니다. [기술 스택/도메인 1줄 요약]

어떤 걸 만들고 싶으세요? 예를 들면:
- 기존 기능 확장 (예: ...)
- 완전히 새로운 기능
- 기술적 개선 (성능, 자동화 등)
```

### 인자가 있는 경우

인자를 초기 주제로 삼되, 바로 구체화 질문으로 들어간다.

### 질문 원칙

**한 번에 1~2개씩, 구체적으로 물어간다.** 포괄적인 질문 금지. 각 턴에서:

1. 이전 답변에서 파악한 것을 1줄로 확인
2. 다음으로 알아야 할 것 1~2개를 구체적으로 질문

질문 축:
- **누구를 위한 것인가?** (사용자 유형, 페르소나)
- **핵심 행동이 뭔가?** (사용자가 실제로 하는 동작)
- **지금은 이걸 어떻게 하고 있나?** (현재 워크플로우, 불편한 점)
- **성공하면 어떤 모습인가?** (기대 결과, 측정 가능한 지표)
- **제약이 있나?** (기술적, 시간적, 비용적)
- **MVP는 뭔가?** (첫 버전에 반드시 있어야 할 것 vs 나중에 해도 되는 것)

### 임시 파일 안내

사용자가 긴 내용을 공유하고 싶어할 때:

```
긴 내용이 있으면 `brainstorm-input.md` 파일에 붙여넣어 주세요.
프로젝트 루트에 만들면 제가 읽겠습니다.
```

`brainstorm-input.md`가 프로젝트에 존재하면 Read로 읽고 내용을 반영한다.

**git 추적 방지:** `.gitignore`에 `brainstorm-input.md`가 없으면 추가한다:
```bash
echo "brainstorm-input.md" >> .gitignore
```

### 전환 판단 — 질문을 멈출 타이밍

아래 4가지를 모두 답할 수 있으면 AI가 먼저 전환을 제안한다:
- 무엇을 만드는가 (what)
- 누구를 위한 것인가 (who)
- 왜 필요한가 (why)
- 첫 버전의 범위 (scope)

```
지금까지 정리하면:
- What: [한 줄]
- Who: [한 줄]
- Why: [한 줄]
- Scope: [한 줄]

이 정도면 방향이 잡힌 것 같은데, 다음 단계로 넘어갈까요?
아니면 더 논의하고 싶은 부분이 있으면 말씀해 주세요.
```

사용자가 동의하면 Step 2로 넘어간다. 추가 논의를 원하면 질문을 계속한다.

## Step 2: 코드베이스 맥락 조사

질문 공세가 끝나면, 합의된 아이디어와 관련된 코드를 조사한다:

1. 아이디어와 관련된 기존 코드 검색 (Grep/Glob)
2. 기존 이슈 확인:
   ```bash
   gh issue list --state open --limit 100 --json number,title,labels
   ```
3. **관련 코드 식별** — 아이디어가 영향을 미칠 파일/모듈
4. **기존 패턴 파악** — 프로젝트가 이미 사용하는 패턴과 자연스러운 접점
5. **제약 조건** — 기술적 제약, 데이터 모델 한계, 외부 의존성

## Step 3: 1원칙 분석 + 5 Whys 정리

수집된 아이디어를 **1원칙 사고(First Principles Thinking)**와 **5 Whys 기법**으로 정제한다.

### 5 Whys — 근본 목적 도출

사용자가 말한 기능 요청에서 출발하여 "왜?"를 반복한다. 대화에서 이미 파악된 why가 있으면 그것을 채우고, 부족한 부분만 깊이 판다. 5회를 기계적으로 채우는 것이 아니라 **근본 목적에 도달하면 멈춘다.**

```
## 🔍 근본 목적 분석

1. 왜 [사용자가 말한 기능]이 필요한가?
   → [답변 — 대화에서 파악했거나 추론]
2. 왜 [답변]이 중요한가?
   → [답변]
3. (근본에 도달할 때까지 계속)
   → **근본 목적: [한 문장]**
```

### 1원칙 분해 — 진짜 필요한 것 도출

근본 목적에서 역방향으로, 기존 가정을 걷어내고 분해한다:

```
## 🧱 1원칙 분해

### 근본 목적
[5 Whys에서 도출한 한 문장]

### 이 목적을 달성하는 데 반드시 참이어야 하는 것
1. [불변 조건 1]
2. [불변 조건 2]

### 가정이었던 것 (제거 가능)
- [당연하다고 생각했지만 실은 선택인 것]

### 최소 구현
불변 조건만 충족하는 가장 단순한 형태:
- [구체적 설명]
```

이 분석을 사용자에게 보여주고, 방향이 맞는지 확인한다.

## Step 4: 방안 제시 + 방향 확정

1원칙 분석 결과를 바탕으로 접근 방안을 제시한다:

```
## 💡 브레인스토밍 결과: [아이디어 요약]

### 근본 목적
[1원칙 분석에서 도출]

### 현재 상태
- [코드베이스에서 파악한 관련 현황]

### 접근 방식

#### 방안 A: [이름]
- 설명: ...
- 장점: ...
- 단점: ...
- 예상 규모: 이슈 N개

#### 방안 B: [이름]
- 설명: ...
- 장점: ...
- 단점: ...
- 예상 규모: 이슈 N개

### 고려 사항
- [기술적 제약이나 트레이드오프]

어떤 방향으로 진행할까요?
```

**사용자 응답을 기다린다.** 사용자가 만족할 때까지 이 단계를 반복한다.

## Step 5: 이슈 분해

사용자가 방향을 확정하면, 구현에 필요한 작업을 이슈 단위로 분해한다:

1. **의존성 순서 파악** — 어떤 이슈가 다른 이슈의 선행 조건인지
2. **적절한 크기로 분할** — 각 이슈가 워커 1개가 처리할 수 있는 규모 (S~M)
3. **라벨 결정** — `feature`, `enhancement`, `refactor` 등

분해 결과를 사용자에게 보여주고 확인받는다:

```
## 이슈 분해 계획

[아이디어] → N개 이슈로 분해

| 순서 | 제목 | 설명 | 의존성 | 라벨 |
|------|------|------|--------|------|
| 1 | ... | ... | — | feature |
| 2 | ... | ... | #1 이후 | feature |

이대로 이슈를 생성할까요?
```

**사용자 확인을 받은 후** Step 6으로 진행한다.

## Step 6: 이슈 생성

라벨 사전체크 후, 확정된 이슈를 **Bash 병렬 호출**로 일괄 생성한다.

```bash
gh label create "feature" --color "0075CA" 2>/dev/null

gh issue create \
  --title "제목" \
  --body "$(cat <<'EOF'
## 배경

[브레인스토밍에서 합의된 맥락]

### 근본 목적
[1원칙 분석에서 도출한 한 문장]

## 요구 사항

- [ ] 요구사항 1
- [ ] 요구사항 2

## 관련 코드

- `파일` — 현재 상태

## 의존성

- 선행: #N (있는 경우)

---
🤖 `/basic-brainstorm`에서 자동 생성됨
EOF
)" \
  --label "라벨"
```

**실행 방식:** 이슈별로 독립된 Bash tool call을 만들어 한 메시지에서 병렬 실행.

## Step 7: 정리 + 결과 요약

### 임시 파일 정리

`brainstorm-input.md`가 존재하면:
```
brainstorm-input.md를 삭제할까요?
```
사용자 확인 후 삭제한다.

### 결과 요약

```
## 브레인스토밍 완료

- 아이디어: [요약]
- 근본 목적: [1원칙 분석 결과]
- 선택 방안: [A/B/...]
- 생성 이슈: N개

| 이슈 | 제목 | 순서 | 라벨 |
|------|------|------|------|
| #601 | ... | 1 | feature |
| #602 | ... | 2 | feature |

다음 단계: `/basic-spec 601 602 603` 으로 스펙 생성
```
