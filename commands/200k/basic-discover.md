> 기회/신기능 발견 → 이슈 생성 | context_tier: 200k

코드베이스를 분석하여 신기능·개선 기회를 발견하고, 이슈로 등록합니다.

인자: `$ARGUMENTS` (선택, 집중 영역 지정 예: `feature`, `dx`, `architecture`. 없으면 전체 관심사 실행)

## Step 0: 학습 데이터 로드

```bash
cat logs/audit-learning.json 2>/dev/null || echo '{"false_positives":[],"true_positives":[]}'
```

이전 감사의 오탐/채택 패턴을 로드한다. 파일이 없으면 빈 상태로 시작 (첫 실행).

## Step 1: 기존 이슈 수집 + 라벨 사전체크

```bash
gh issue list --state open --limit 100 --json number,title,labels
gh label list --json name
```

1. 이슈 목록을 이후 모든 발견과 대조한다. 이미 이슈가 있는 아이디어는 건너뛴다.
2. Step 5에서 사용할 라벨(`feature`, `enhancement`, `dx`, `performance`, `refactor` 등)이 존재하는지 확인한다. 없는 라벨은 `gh label create`로 미리 생성한다.

## Step 2: 프로젝트 컨텍스트 파악

코드베이스의 전체 구조를 빠르게 파악한다:
1. `package.json` (또는 해당 언어의 매니페스트)에서 기술 스택, 의존성, 스크립트 확인
2. 디렉토리 구조 파악 (`ls`, Glob)
3. README에서 프로젝트 목적·도메인 파악

이 정보를 Agent 프롬프트에 포함하여 프로젝트 맥락에 맞는 제안을 유도한다.

## Step 3: 관심사별 Explore Agent 병렬 실행

3개 관심사 × 관심사당 1개 Explore Agent(model: **sonnet**)를 **병렬로** 실행한다. `$ARGUMENTS`로 영역이 지정된 경우 해당 agent만 실행한다.

**agent 수 조정:** 대상 파일이 많아 단일 agent의 컨텍스트로 부족할 경우, 해당 관심사를 디렉토리별로 분할하여 복수 agent를 실행한다.

### Agent 1: 기능 확장 기회 (feature)

프로젝트의 주요 기능 코드를 탐색하여 **빠져 있는 기능, 불완전한 기능, 사용자 경험 향상 기회**를 찾는다. 프로젝트 유형에 맞게 탐색 관점을 조정한다.

### Agent 2: DX + 자동화 기회 (dx)

프로젝트의 빌드 설정, 스크립트, CI, 타입 정의를 탐색하여 **개발 경험·자동화 개선 기회**를 찾는다. 프로젝트 유형에 맞게 탐색 관점을 조정한다.

### Agent 3: 아키텍처 진화 기회 (architecture)

프로젝트의 전체 구조, 모듈 간 의존성을 탐색하여 **아키텍처 개선·확장성 확보 기회**를 찾는다. 프로젝트 유형에 맞게 탐색 관점을 조정한다.

### Agent 공통 지시

각 Agent에게 전달할 지시:
```
프로젝트 컨텍스트: [Step 2에서 파악한 기술 스택, 도메인, 구조 요약]

발견 항목을 아래 형식으로 반환하라:

### 기회 1: [제안 요약]
- 위치: `파일:라인` (관련 코드가 있는 경우)
- 영역: feature / dx / architecture
- 근거: 확인됨 / 추정
- 임팩트: 높음 / 중간 / 낮음
- 설명: 무엇을 왜 추가/개선하면 좋은지

### 기회 2: ...

규칙:
- 근거 "확인됨": 코드를 직접 읽고 기회를 특정한 경우 (파일:라인 필수)
- 근거 "추정": 코드 패턴상 기회가 있을 것 같지만 완전히 확인하지 못한 경우
- 이미 구현된 것은 제안하지 않는다
- 코드를 읽고 구체적 근거가 있는 제안만 한다
- 버그/취약점은 여기서 다루지 않는다 (audit 영역)

## 이전 감사 학습 데이터 (Step 0에서 로드한 내용이 있을 때만 포함)

### 오탐 패턴 (제안하지 마라)
{audit-learning.json의 false_positives에서 해당 agent 관심사만 필터링하여 나열}

### 잘 찾은 패턴 (더 찾아라)
{audit-learning.json의 true_positives에서 해당 agent 관심사만 필터링하여 나열}
```

## Step 4: 발견 통합 + 이슈 그룹핑

Agent 결과를 수집하여:

1. **중복 제거** — 같은 영역/파일을 여러 agent가 제안한 경우 병합
2. **기존 이슈 대조** — Step 1에서 수집한 open 이슈와 비교, 이미 있으면 스킵
3. **오탐 패턴 대조** — Step 0에서 로드한 `audit-learning.json`의 `false_positives`와 비교. 제안 요약이 기존 오탐 pattern과 유사하고 file_pattern이 일치하면 자동 스킵. 매칭 시 count +1, last_seen 갱신.
4. **그룹핑 판단**:
   - 관련된 제안 → 하나의 이슈로 그룹
   - 독립적인 제안 → 단독 이슈
5. **자동 필터링 — 이슈 생성 대상 결정**:
   - 근거 "확인됨" + 임팩트 높음/중간 → **자동 생성**
   - 근거 "확인됨" + 임팩트 낮음 → 본체가 2차 판단 → 의미 있으면 생성, 아니면 요약에만 기록
   - 근거 "추정" → 본체가 해당 코드를 직접 Read로 확인 → 진짜면 생성, 아니면 버림
   - 기존 이슈와 겹침 → **생성하지 않음** (스킵 로그에 기록)

## Step 5: 이슈 생성

필터링을 통과한 항목에 대해 **Bash 병렬 호출**로 이슈를 일괄 생성한다.
에이전트를 띄우지 않고 본체가 직접 `gh issue create`를 병렬 Bash tool call로 실행한다.

```bash
gh issue create \
  --title "제목" \
  --body "$(cat <<'EOF'
## 제안

[발견한 기회 설명]

## 현재 상태

- `파일:라인` — 현재 코드가 어떻게 되어 있는지

## 기대 효과

[이 기능/개선이 가져올 가치]

## 구현 힌트

[대략적인 접근 방향 — spec에서 상세화]

---
🤖 `/basic-discover`에서 자동 생성됨
EOF
)" \
  --label "라벨"
```

라벨 매핑:
- 신기능 → `feature`
- 기존 기능 개선 → `enhancement`
- DX/자동화 → `dx`
- 성능 → `performance`
- 아키텍처 → `refactor`

**실행 방식:** 이슈별로 독립된 Bash tool call을 만들어 한 메시지에서 병렬 실행.

## Step 6: 결과 요약

```
## 디스커버 완료

- 탐색 영역: [전체 / 지정된 영역]
- 발견 기회: N개
- 생성 이슈: M개
- 스킵 (기존 이슈): K개
- 보류: L개

| 이슈 | 제목 | 영역 | 라벨 |
|------|------|------|------|
| #501 | ... | feature | feature |
| #502 | ... | dx | enhancement |

### 오탐 판정 (추정 → 버림)
| 제안 | 판정 이유 |
|------|----------|
| ... | ... |

다음 단계: `/basic-spec 501 502 503` 으로 스펙 생성
```

## Step 7: 학습 데이터 갱신

감사 결과를 `logs/audit-learning.json`에 반영한다:

1. **새 오탐** (Step 4에서 버린 항목 중 신규 패턴) → `false_positives`에 추가 (`pattern`, `file_pattern`, `agent`, `reason`, `count: 1`, `last_seen`)
2. **기존 오탐 반복** → `count` +1, `last_seen` 갱신
3. **새 채택** (이슈 생성된 항목) → `true_positives`에서 유사 패턴 검색. 있으면 `adopted_count` +1, 없으면 새로 추가
4. **만료 정리** → `last_seen`이 90일 이상 지난 `false_positives` 항목은 제거 (코드 변경 후 재검사하도록)
