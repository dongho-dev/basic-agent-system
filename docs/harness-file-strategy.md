# 하네스 엔지니어링 — 파일 전략 핵심 원칙

## 1. 커맨드 파일 분리는 필수

하네스가 커맨드를 실행할 때 **로드된 프롬프트 = 실행할 작업**이어야 한다. 12개 커맨드가 한 파일이면 LLM이 "어떤 걸 실행해야 하지?"부터 판단해야 하고, 이건 하네스 엔지니어링의 근본 원칙에 위배된다.

> **LLM에게 판단을 시키지 말고, 하네스가 판단을 끝내고 LLM에게 실행만 시켜라.**

### 분리의 세 가지 이점

| 축 | 효과 |
|----|------|
| 토큰 예산 | 불필요한 커맨드가 컨텍스트를 차지하지 않음 → 코드 읽기에 더 많은 공간 |
| 지시 정밀도 | 경쟁하는 지시가 없어 hallucination/혼동 확률 감소 |
| 속도 | prefill 토큰 감소 → first-token latency 개선 |

### 커맨드 간 중복은 허용

Claude Code 커맨드에는 `include`/`partial` 메커니즘이 없다. 현재 규모(12파일, 5-11KB)에서는 **중복 유지 + self-contained 파일**이 빌드 파이프라인보다 낫다. 파일 수가 50개+, 중복이 수십 KB가 되면 빌드 스크립트를 고려.

## 2. 1M vs 200k — 병목이 다르다

### 200k 티어

토큰 부족이 실제로 온다. 커맨드 프롬프트 크기를 줄이는 것이 직접적 레버리지.

### 1M 티어

토큰 자체는 거의 부족하지 않는다. 병목은:

- **비용** — 커맨드가 유발하는 agent 결과 누적으로 input 토큰 비용 상승
- **속도** — tool 결과 누적이 prefill을 키움
- **주의력 희석** — 컨텍스트 길어지면 후반 지시에 대한 attention 저하

→ 1M에서 최적화 방향은 파일 크기가 아니라 **실행 중 컨텍스트 팽창 제어** (agent 반환 크기 제한, 중간 결과를 파일로 쓰기 등).

## 3. 개별 프로젝트 코드 구조 — LLM 효율에 영향

커맨드가 실행되는 대상 프로젝트의 구조가 하네스 효율을 결정한다.

### 레버리지 순위

| 순위 | 요소 | 영향도 | 전통 원칙과 |
|------|------|--------|------------|
| 1 | `permissions.deny`로 불필요 파일 제외 | 높음 | 해당 없음 (신규) |
| 2 | 파일 네이밍 컨벤션 (파일명만으로 맥락 파악) | 높음 | 일치 |
| 3 | 코드 중복 제거 (DRY) | 높음 | 일치 |
| 4 | 파일 크기 200-500줄 유지 | 중간 | 일치 |
| 5 | 핵심 타입을 접근 가능한 위치에 모으기 | 중간 | 약간 다름 |
| 6 | 추상화 깊이 인지 (간접 참조 = 컨텍스트 비용) | 중간 | 갈림 |
| 7 | 테스트 파일 도메인별 분리 | 낮음 | 일치 |

### 파일 제외 방법 (2025 기준)

- **`.gitignore`** — Claude Code가 자동 존중. node_modules, dist 등은 기본 제외.
- **`permissions.deny`** — `.claude/settings.json`에서 Read를 명시적 차단. lock 파일, 빌드 산출물, 대형 설정 등에 사용.
- **`.claudeignore`** — 네이티브 미지원. 커뮤니티 훅(claude-ignore)으로 비공식 구현 가능.

### 전통 원칙과 갈리는 지점

**추상화 깊이**: Controller → Service → Repository → DataMapper → QueryBuilder 같은 깊은 레이어링은 전통적으로 좋은 설계지만, LLM은 한 흐름을 이해하려고 5개 파일을 읽어야 한다. 간접 참조(indirection)는 컨텍스트 비용.

**타입 집중 배치**: 도메인별 타입 분산보다, 핵심 타입을 한 곳에 모아두면 LLM이 타입 파일 하나로 도메인 모델을 빠르게 파악. 단, 1000줄 넘으면 분리.

> **전통적 "좋은 코드 구조"의 80%는 LLM 효율과 일치. 갈리는 20%는 "간접 참조 비용"과 "탐색 가능성(discoverability)".**

## 4. 탐색 트레이드오프 (결정: 현행 유지)

파일을 작게 쪼갤수록 Read 비용은 줄지만, Glob/Grep 탐색 비용이 늘어난다.

**결론:** Grep(ripgrep)이 이미 인덱스 역할을 충분히 한다. 분산 CLAUDE.md나 심볼 인덱스 자동 생성은 절약분 대비 복잡도가 높아 도입하지 않는다. 네이밍 컨벤션 + 현행 Grep으로 충분.

## 5. 변경 결합도 Agent (결정: audit에 추가)

기존 audit 커맨드(backend/frontend/fullstack)에 Agent C2를 추가한다.

```
C2: 변경 결합도 + 응집도

1. git log에서 co-change 패턴 수집 (Step 1에서 gh issue list와 함께)
2. 같은 파일 안에서 변경 이유가 다른 블록 → 분리 후보
3. 다른 파일인데 항상 같이 바뀌는 것 → 합체 후보
4. 코드를 읽고 의미적 교차 검증
5. git 실증 + 의미적 추론 일치 시에만 이슈 생성
```

기준: 이슈 수나 파일 크기가 아니라, **수정할 때 관련 없는 코드가 같이 영향받는지**. MIS 최적화는 코드 구조 결정의 부산물이어야지 목적이 되면 안 된다.

## 6. Worker 실수 방지 — 구조적 검증 게이트 (방향 확정: 구현 대기)

프롬프트로 "~하라"고 지시해도 agent가 안 따르는 건 학습으로 해결되지 않는다. pipeline-cli.mjs가 phase 순서를 코드로 강제한 것처럼, **검증 게이트를 확장**하여 구조적으로 차단해야 한다.

| 문제 | 구조적 강제 |
|------|-----------|
| 리뷰 생략 | CLI가 review 결과 없이 merge phase 진입 차단 |
| Closes #N 누락 | PR body에 키워드 없으면 merge 차단 |
| 파일 매핑 누락 | spec body의 Write 파일과 agent 보고 파일 수 불일치 시 경고 |

원칙: LLM 판단에 의존하지 않고, 코드가 차단한다.
